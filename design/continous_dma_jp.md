# 連続DMAパイプラインによる低レイテンシHILS I/O設計

[English Version](./continous_dma_en.md)


## 設計課題

　HILS (Hardware-in-the-Loop Simulation) は、ECU実機とプラントモデルを接続し、制御アルゴリズムの検証を行うための重要な環境である。HILSでは、ECUとプラントモデルが並列に実行され、一定周期 ΔT ごとにインターフェースボードを介してデータを交換する必要がある(図1)。

![HILSのブロック図](./continous_dma/block_diagram.png)

　ΔTはプラントモデルの時定数によって決まり、特にモータ電流制御やインバータ制御などの高速電力系では、数μs〜数十μs程度の短周期動作が要求される。このため、HILS がリアルタイムに成立するためには、1周期内に I/O 処理とプラント計算が完了している必要がある。これは次式で表される。

`ΔT > T_io + T_compute`

　しかし実際のHILSシステムでは、I/Oレイテンシ T_io が ΔT の10%〜30%[1]、場合によってはそれ以上を占めることがあり[2]、短周期化の大きな制約となる。特に、CPUがDMAの設定や転送完了検出に介在する構成では、ソフトウェア処理に起因するオーバーヘッドが無視できない。

　一般的な構成では、I/Oレイテンシは以下のように分解できる：

`T_io ≃ 2 × (T_setup + T_transfer + T_sync)`

　ここで、T_setup はDMAの設定および開始処理、T_sync は転送完了の検出であり、いずれもCPUによるPIOアクセスや割り込み処理に依存する。このCPU介在がレイテンシおよびジッタの主要因となる。

　一方で、I/O処理とプラントモデルをすべてFPGAに実装することでこれらの問題を回避することも可能であるが、モデル変更に対する柔軟性が低く、開発コストも高い。このため、汎用CPU環境を維持したまま、I/Oレイテンシのみを削減する手法が求められていた。

## 解決アプローチ


　従来方式(図2)では、DMAごとにCPUが介在し、T_setupおよびT_syncが発生する。

![従来手法のタイミングチャート](./continous_dma/time_chart_conventional.png)

　これに対して本設計(図3)では、DMA転送をCPUから完全に切り離し、インターフェースボード上のFPGAが自律的にDMAを実行する構成を採用した。

![連続DMAパイプラインのタイミングチャート](./continous_dma/time_chart_continous.png)



　DMAディスクリプタは初期化時に一度だけ設定し、HILS実行中は再設定を行わない。以降の周期では、FPGAがバスマスタとして自律的にDMA転送を開始する。

　転送の開始可否は、転送対象バッファの先頭に配置されたタイムスタンプによって判定する。DMAはメモリを先頭から読み出し、最初のバーストでタイムスタンプを取得する。タイムスタンプが更新されていない場合、後続の転送を行わずに中断し、再度DMA転送を試行する。一方、タイムスタンプが更新されている場合にのみ、後続データの転送を実行する。

　この構成により、従来CPUが担っていたDMA開始処理および転送完了検出処理を不要とし、I/Oレイテンシは以下のように近似される：

`T_io ≃ 2 × (T_trial + T_transfer)`

　ここで、T_trial はタイムスタンプ確認のためのDMA試行に要する時間である。

　タイムスタンプは64ビットの単調増加カウンタとし、各周期ΔTごとに1インクリメントする。本設計では、このタイムスタンプをデータ更新のコミットポイントとして扱う。

　CPU側では、まず全データを更新し、その後にタイムスタンプを更新する。さらに、書き込み順序が入れ替わらないよう、write memory barrier（store-store barrier）等により書き込み順序を保証する。また、DMA対象バッファはキャッシュコヒーレントもしくはノンキャッシュ領域とし、非コヒーレント環境ではキャッシュフラッシュを行う。

　このようにデータ更新完了後にのみタイムスタンプが更新されるため、DMAはタイムスタンプ更新済みのデータのみを観測することになる。タイムスタンプ未更新の場合は転送を中断するため、更新途中のデータが後続バーストとして読み出されることは抑制される。

　また、DMAのリトライは最小バースト転送を伴う短時間のDMA試行であるが、バス使用権の獲得および最小バースト転送を伴うため、一定時間バスを占有する。このため、リトライ回数が増加するとバス利用効率が低下する。

　そこで、本設計ではDMA転送開始タイミングを動的に補正する仕組みを導入した。具体的には、前周期における転送成功時刻との差分に基づき、次周期のDMA開始タイミングを調整する。これにより、タイムスタンプ更新直後にDMAを開始できるようになり、リトライ回数を最小化することができる。

　初期状態では、DMAはT_setupおよびT_sync相当の時間だけ早く開始されるため、リトライが発生するが、補正が進むにつれてT_trialは0近傍に収束し、定常状態ではほぼリトライなしで転送が行われる。


## 制約およびトレードオフ

　本設計は、DMAの設定および転送完了検出に伴うソフトウェアオーバーヘッドを排除するものであり、これらのオーバーヘッドが支配的となる小規模データ転送において特に効果が大きい。一方で、転送データサイズが大きくなると、バス上でのデータ転送時間 T_transfer が支配的となり、相対的な改善効果は小さくなる。

　また、本方式ではDMAのリトライによりバス占有が発生するため、他のバスマスタとの競合が存在する環境では、DMA開始タイミングの最適化が重要となる。

　さらに、データ一貫性の確保のために、メモリの可視性および書き込み順序に関する前提条件（キャッシュコヒーレンシ、メモリバリア）が満たされている必要がある。

## 追記

　本設計は、私が実際にHILSシステムのI/Oレイテンシ改善を目的として設計・実装したものであり、33MHz/32bit PCIバスおよびリアルタイムLinux（PREEMPT_RT）環境において評価を行った。その結果、6ワード転送（往復12ワード）の条件で、従来約62μsであったI/Oレイテンシを約5.6μsまで削減し、約90%のレイテンシ低減（約62μs → 約5.6μs）を達成した(図4)。また、ジッタは1μs以内に抑制された。

![レイテンシの比較](./continous_dma/latency_comparison.png)

　本設計はHILSに限らず、周期的にデータ転送を行う任意のリアルタイムシステムに適用可能である。本設計思想および実装方式は、関連特許[WO20110060852](https://patents.justia.com/patent/20110060852)として権利化されている。
